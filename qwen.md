## 🛠 Технический стек и инструменты

### Xcode MCP (Обязательно к использованию)
Для взаимодействия с проектом **необходимо** использовать инструменты Xcode MCP. Это обеспечивает синхронизацию между кодом и структурой проекта Xcode.

**Основные юзкейсы Xcode MCP:**
- **XcodeLS**: Исследование структуры групп и файлов проекта (не путать с файловой системой).
- **XcodeRead / XcodeWrite**: Чтение и запись файлов с автоматическим обновлением `.pbxproj`.
- **XcodeUpdate**: Точечное редактирование содержимого файлов.
- **BuildProject**: Проверка компилируемости после изменений.
- **RunAllTests / RunSomeTests**: Валидация изменений через систему тестирования.
- **RenderPreview**: Проверка UI-компонентов SwiftUI через Swift Previews.
- **XcodeListNavigatorIssues**: Мониторинг ошибок компиляции и предупреждений в реальном времени.

### Архитектура
- **Clean Architecture / MVVM**: Обязательное разделение ответственности.
    - **Data Layer**: GRDB.swift для CRUD операций с SQLite.
    - **Domain Layer**: swift-fsrs библиотека для расчёта интервалов.
    - **Presentation Layer**: SwiftUI представления + ViewModel с `@Observable`.

## 🧠 Алгоритм интервального повторения (FSRS)

### Почему FSRS вместо SM-2
- **Снижение нагрузки на 25%** при том же уровне удержания.
- **Устранение "Ада легкости"** (Ease Hell) — порочного круга частых повторений.
- **Трёхкомпонентная модель памяти (DSR)**:
  - **Difficulty (D)**: Внутренняя сложность билета.
  - **Stability (S)**: Сила сохранения памяти (время до забывания).
  - **Retrievability (R)**: Вероятность извлечения в текущий момент.

### Бинарная система оценки
- **Верно** → Grade 3 (Good) — алгоритм пересчитывает прирост стабильности.
- **Неверно** → Grade 1 (Again) — алгоритм регистрирует провал (lapse), снижает извлекаемость.
- **Преимущества**: Устраняет когнитивные искажения (JOL), снижает усталость от решений на 13-15%.

### Целевые параметры
- **Desired Retention**: 85-90% (оптимум между нагрузкой и эффективностью).
- **Предэкзаменационный режим**: Массированное тестирование по 300-500 карточек/день с временной приостановкой FSRS.

## 🗄 Архитектура базы данных (SQLite)

### Требования
- **SQLite** через GRDB.swift (не SwiftData/Core Data).
- **Нормализованная схема** с тремя основными таблицами.

### Схема БД

#### Таблица `Notes` (Контент)
| Поле | Тип | Описание |
| --- | --- | --- |
| note_id | UUID | Уникальный идентификатор |
| chapter_id | Int | Привязка к главе ПДД (1-26) |
| question_text | Text | Текст вопроса |
| correct_answer | Text | Текст правильного ответа |
| distractors | Text (JSON) | Массив неправильных вариантов |
| image_reference | Text | Путь к изображению или URL |

#### Таблица `Cards` (Состояние планировщика)
| Поле | Тип | Описание |
| --- | --- | --- |
| card_id | UUID | Уникальный идентификатор |
| note_id | UUID (FK) | Внешний ключ к Notes |
| due_date | DateTime (UTC) | Дата следующего повторения |
| stability | Double | Текущая стабильность (FSRS) |
| difficulty | Double | Текущая сложность (FSRS) |
| retrievability | Double | Вероятность припоминания |
| state | Enum | Новая / В изучении / Выучена |
| lapses | Int | Количество забываний |

#### Таблица `Revlogs` (Журнал повторений, Append-only)
| Поле | Тип | Описание |
| --- | --- | --- |
| log_id | UUID | Уникальный идентификатор лога |
| card_id | UUID (FK) | Внешний ключ к Cards |
| review_datetime | DateTime (UTC) | Точный timestamp ответа |
| grade | Int | Оценка: 1 (ошибка) или 3 (успех) |
| time_taken | Int | Время ответа в миллисекундах |
| pre_stability | Double | Стабильность до ответа |
| pre_difficulty | Double | Сложность до ответа |

## 🏗 Архитектурные слои (Clean Architecture)

### 1. Data Layer
- **GRDB.swift**: Обёртка над SQLite для CRUD операций.
- **Репозитории**: Абстракция над прямыми SQL-запросами.
- **Миграции**: Версионирование схемы БД.

### 2. Domain Layer
- **swift-fsrs**: Библиотека для расчёта интервалов (Swift Package Manager).
- **Scheduler Protocol**: `scheduler.review_card(card, grade)` → обновлённая Card + ReviewLog.
- **Entities**: Чистые модели без зависимостей от UI/DB.
- **Use Cases**: Бизнес-логика (например, `CalculateNextInterval`, `GetDueCards`).

### 3. Presentation Layer (SwiftUI)
- **ViewModel**: `@Observable` классы, вызывают Domain Layer.
- **Views**: Только отображение, без бизнес-логики.
- **Fisher-Yates Shuffle**: Обязательное перемешивание вариантов ответов перед рендерингом.
- **Тактильный отклик**: Core Haptics при ответе.
- **Немедленная обратная связь**: При ошибке — блокировка, подсветка, пояснение из ПДД.

- **3000 билетов ПДД РБ**, разбитых по **26 главам**.

- **Фильтрация**: Возможность изучать отдельные главы (например, только Глава 13 — "Проезд перекрёстков").
