# TrafficMaster iOS MVP Tasks

Дата: 2026-05-16

## 1. Целевое видение приложения

### 1.1 Вкладки MVP (5 штук)
- Сегодня
- Разделы
- Зачёты
- Экзамен
- Профиль

Примечание: отдельную вкладку `Ошибки` в MVP не добавляем. Ошибки встраиваются в `Сегодня` и `Разделы`.

### 1.2 Основной пользовательский сценарий
- Пользователь проходит вопросы по разделам.
- Каждый ответ отправляется в SM-2 backend.
- Backend рассчитывает следующий показ карточки.
- iOS ежедневно собирает тренировку до 20 минут из:
  - просроченных повторений;
  - ошибок;
  - слабых вопросов;
  - новых вопросов из текущего раздела;
  - короткого экзаменационного блока.

Целевой UX: `Разделы ПДД -> зачёты по разделам -> комплексные зачёты -> экзамен`, с внутренней SRS-логикой.

## 2. Модель прогресса

### 2.1 Два независимых состояния
- `coverage state`: видел ли пользователь вопрос хотя бы один раз.
- `SM-2 state`: когда и как повторять вопрос.

### 2.2 Источник истины
Backend хранит SM-2 состояния карточки:
- `new`
- `learning`
- `review`
- `relearning`

В iOS эти состояния показываются через UI-лейблы:
- Новый
- На изучении
- Слабый
- Ошибка
- Закрепляется
- Выучен

Важно: UI-лейблы не заменяют алгоритм, это только интерпретация backend-состояний.

## 3. Маппинг ответов в SM-2 ReviewRating

Backend значения:
- `AGAIN = 1`
- `HARD = 2`
- `GOOD = 3`
- `EASY = 4`

Маппинг для теста ПДД (2-5 вариантов, один правильный):

| Ситуация пользователя | Текст в UI | Rating в backend |
|---|---|---|
| Ответ неверный | Неверно | AGAIN (1) |
| Ответ верный, но пользователь угадал | Угадал | AGAIN (1) |
| Ответ верный, но были сомнения | Сомневался | HARD (2) |
| Ответ верный уверенно | Знал | GOOD (3) |
| Ответ верный мгновенно, вопрос очевидный | Легко | EASY (4) |

Ключевое правило: `Угадал` не считается знанием и не должен повышать карточку как `GOOD`.

## 4. Текущий статус реализации (после удаления марафона)
